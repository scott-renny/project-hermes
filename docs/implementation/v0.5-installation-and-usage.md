# Project Hermes v0.5 Installation and Usage

## Status

This guide documents the supported v0.5.0 workstation-framework workflow. The
release provides validated component modules and a unified declarative profile.
It does not yet provide one command that applies the entire workstation profile.

## Supported Environment

- Windows 11 Home or Pro
- PowerShell 7.0 or later
- Git
- WinGet for package-profile auditing and installation
- Pester 6 for repository validation

Run commands from the Project Hermes repository root unless stated otherwise.

## 1. Clone the Repository

```powershell
git clone https://github.com/scott-renny/project-hermes.git
cd project-hermes
```

## 2. Validate Prerequisites

```powershell
.\scripts\diagnostics\Test-HermesEnvironment.ps1 -Detailed
```

Resolve reported prerequisite failures before applying workstation settings.

## 3. Run the Complete Test Suite

```powershell
Invoke-Pester `
    -Path @(
        '.\modules'
        '.\tests'
    ) `
    -Output Detailed
```

The validated v0.5.0 baseline contains 314 passing tests with no failures.

## 4. Review the Unified Profile

```powershell
$workstationProfile = Import-PowerShellDataFile `
    -LiteralPath .\configs\profiles\hermes-workstation-base.psd1

$workstationProfile | Format-List
$workstationProfile.Components | Format-Table
```

The unified profile references ten component modules and their version-controlled
desired-state files. Review its enabled, required, and execution-order fields
before changing the workstation.

## 5. Preview a Component

Import the component module and desired-state file. This example previews the
native Windows appearance baseline:

```powershell
Import-Module `
    .\modules\workstation\windows\Hermes.Windows.psd1 `
    -Force `
    -ErrorAction Stop

$configuration = Import-PowerShellDataFile `
    -LiteralPath .\configs\windows\hermes-visual-base.psd1

Test-HermesWindowsConfiguration `
    -Configuration $configuration |
    Format-List

Test-HermesWindowsSettings `
    -Configuration $configuration |
    Format-List

Set-HermesWindowsSettings `
    -Configuration $configuration `
    -WhatIf
```

Do not continue if configuration validation fails or the preview includes an
unexpected change.

## 6. Apply a Component

After reviewing the preview:

```powershell
$result = Set-HermesWindowsSettings `
    -Configuration $configuration `
    -Confirm:$false

$result | Format-List
```

State-changing modules create backups by default where practical and verify the
result after application. Preserve the returned backup path.

## 7. Install the Managed PowerShell Profile

The PowerShell profile loads completed Hermes modules in future PowerShell 7
sessions:

```powershell
Import-Module `
    .\modules\workstation\powershell\Hermes.PowerShell.psd1 `
    -Force `
    -ErrorAction Stop

$configuration = Import-PowerShellDataFile `
    -LiteralPath .\configs\powershell\hermes-powershell-base.psd1

Set-HermesPowerShellSettings `
    -Configuration $configuration `
    -WhatIf
```

Apply only after reviewing the preview. If Windows Controlled Folder Access
blocks the profile, keep the protection enabled and explicitly allow the trusted
PowerShell 7 executable before retrying.

## 8. Package Profile Safety

`Hermes.Winget` audits approved packages and installs only missing packages from
selected profiles. It does not uninstall software or perform blanket upgrades.

```powershell
Import-Module `
    .\modules\workstation\winget\Hermes.Winget.psd1 `
    -Force `
    -ErrorAction Stop

$configuration = Import-PowerShellDataFile `
    -LiteralPath .\configs\winget\hermes-winget-base.psd1

Test-HermesWingetPackages `
    -Configuration $configuration |
    Format-List

Install-HermesWingetPackages `
    -Configuration $configuration `
    -WhatIf
```

## 9. Generated Data

Backups, inventories, logs, and runtime output remain under ignored repository
paths such as `exports/` and `logs/`. Review generated data before sharing it and
never commit credentials, tokens, personal paths, or unreviewed inventories.

## 10. Recovery

Use the owning module's restore command with a verified backup path. Recovery is
component-specific in v0.5.0; unified restore orchestration is planned for a later
release.

Consult the component README before restoring settings.

## Known v0.5.0 Boundaries

- No one-command unified apply workflow
- No unified restore workflow
- No automatic drift-repair schedule
- Windhawk and Rainmeter configuration remain future visual-integration work
- WSL, Docker, Python, Node.js, and broader developer provisioning remain v0.6 work
